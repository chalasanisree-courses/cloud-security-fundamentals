Module 3  ·  Enterprise Security Architecture

The Castle and the Moat

Module 3 Enterprise architecture Defense in depth
Learning Objectives
  • Describe the three-tier enterprise architecture: client machines, servers, and the networking that connects them
  • Identify the security equipment that forms a network perimeter — firewalls, IDS/IPS, and endpoint protection
  • Explain the castle-and-moat model and why enterprises separate the internet from the internal network
  • Trace how inbound and outbound traffic is inspected in a DMZ
  • Map the five core Functions of the NIST Cybersecurity Framework onto a real-world response
  • Recognize that moving to the cloud preserves — not replaces — this architecture

1 From one connection to the whole bank

In the previous module we secured a single connection — a customer reaching a bank's web server, protected with encryption and a digital certificate that verifies the bank is really the bank. That solved confidentiality on the wire.

But now let's peek inside the bank. Behind that one web server is an entire enterprise architecture — many machines, many networks, and a whole class of equipment whose only job is to keep attackers out. That architecture is what this module is about.

2 The three-tier application

A primitive enterprise looks simple: employees have client machines (laptops, desktops) which they use to reach applications running on servers. Picture a bank:

  • A bank manager's laptop is used to take mortgage applications.
  • Those applications are processed by an application server.
  • Your mortgage data is stored on a database server.
  • You, the customer, check your status by reaching the web server.

This is the classic three-tier architecture: a web tier, an application tier, and a database tier. It's the backbone of almost every enterprise application — but it's only the bones. Let's add flesh.

The three-tier application

The three tiers
Web server takes requests from the outside → application server runs the business logic → database server stores the data. Each tier has a different job, and — as we'll see in the next note — lives on its own network for good reason.

3 Add the plumbing — and the guards

Clients and servers don't connect by magic. Networking equipment — switches and routers — interconnects everything and provides the path to the internet. And once you're connected to the internet, you need a new class of equipment whose entire purpose is protection: security equipment like firewalls and IDS/IPS.

Two kinds of infrastructure, two kinds of security
An enterprise has computing hardware (clients and servers) and networking hardware (switches, routers). Security mirrors that split: network security (firewall, IDS, IPS) guards the wires, and endpoint security (antivirus on clients and servers) guards the machines.

4 The castle-and-moat model

Stack all that security equipment together and it does one thing: it establishes a strong network perimeter that separates the dangerous internet from the trusted enterprise network. The internet is full of malware, viruses, and attackers probing to get in; the perimeter is what gates who gets through.

This is the castle-and-moat architecture. The parallel is exact — click to map it:

Castle ↔ Enterprise

The point of the walls is to protect the crown jewels — in an enterprise, that's the corporate data: credit card numbers, financial records, health information. Everything inside the perimeter exists to keep that data out of the wrong hands.

5 The DMZ — a guarded landing zone

You can't just build walls and seal yourself off — an enterprise has to talk to the outside world. Legitimate customers need to reach the bank's website. So how do you let good traffic in without trusting it blindly? You give it a landing zone: the DMZ (demilitarized zone).

External-facing traffic lands in the DMZ first, gets inspected, and only the good traffic is forwarded to the internal servers. Step through three common flows:

Traffic through the DMZ

What lives in the DMZ
The web server (for inbound customer traffic), the secure web gateway / SWG (inspects employees' outbound web traffic on the way back), and the email gateway (scans inbound email before it reaches the mail server). The DMZ is inside the perimeter, but it's the only place external traffic is allowed to touch first.

6 Your house is a castle too

Forget the castle for a moment — think of your house. You have locks and window bars to keep intruders out (that's your firewall and antivirus). But you also have cameras, because even with locks you want to know if something's wrong (that's intrusion detection). If a camera catches a burglar, you call the security company (response), and afterward you replace what was stolen (recovery).

That sequence — figure out what to protect, protect it, watch for trouble, respond, recover — is exactly how professionals structure security. It has a name.

7 The NIST framework

The NIST (National Institute of Standards and Technology) Cybersecurity Framework organizes security into Functions — five core ones that follow the sequence you just walked through at home. Most security products on the market are built to serve one or more of them. Click each Function:

The five core NIST Functions

CSF 2.0 added a sixth Function: Govern
In 2024 NIST released CSF 2.0, which adds Govern — the organizational layer that wraps around the other five: who owns security, what the policies are, how risk decisions get made. The five Functions above are still the operational lifecycle; Govern is the management structure that directs them. If you look up the framework today, you'll see a six-part wheel.

8 Moving to the cloud changes the address, not the architecture

Here's the punchline of the whole module. When you move to the cloud, your servers might move to AWS, or you might drop some servers entirely and adopt SaaS. Either way, the architecture stays the same: you still create an isolated environment, you still separate your infrastructure from the outside world, and you still protect what's inside.

The walls become security groups and VPCs, the moat becomes cloud networking, but the shape — perimeter, layers, identify-protect-detect-respond-recover — carries over. That's why this model is worth learning once and reusing everywhere, which is exactly what the rest of this course does.

Remember

An enterprise is a three-tier app (web → application → database) wrapped in security equipment that forms a network perimeter — the castle-and-moat model. External traffic lands in the DMZ to be inspected before it's trusted. Network security guards the wires; endpoint security guards the machines. The NIST framework — Identify, Protect, Detect, Respond, Recover (plus Govern, added in CSF 2.0) — names the full lifecycle. And moving to the cloud keeps this architecture intact.
