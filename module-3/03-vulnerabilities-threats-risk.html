Vulnerabilities, Threats & Risk

Module 3 Vulnerabilities CVE · CVSS
Learning Objectives
  • Restate the relationship between threat, vulnerability, and risk
  • Classify a weakness as a configuration, software, or design/architectural vulnerability
  • Read a CVE entry — its identifier and its CVSS severity score
  • Explain the Shellshock vulnerability and why remote code execution is so dangerous
  • Trace the vulnerability lifecycle from introduction to patch, and the window of exposure
  • Contrast attacker reconnaissance with defender scanning

1 Nothing bad happens until a vulnerability is exploited

Recall the relationship from the last note. A vulnerability is how harm becomes possible — a weakness in the system. A threat is what causes the harm — an actor exploiting that weakness. A vulnerability sitting alone causes nothing. Risk is the point where a real threat can reach a real weakness. Everything in this note is about finding and closing weaknesses before a threat finds them first.

2 Three types of vulnerability

Weaknesses fall into three broad families. Click each to see what it looks like in a real enterprise:

Where weaknesses come from

Case study — Target, 2013
Target was an architectural failure. The network that faced HVAC vendors was reachable from the point-of-sale systems — a flat network with no real segmentation, and vendors holding far more access than they needed. That's the castle-and-moat trap: once inside the walls, attackers had free range. Proper segmentation and least privilege would have contained the breach.

3 How do you even know a vulnerability exists?

You don't discover most weaknesses yourself — you look them up. NIST publishes the National Vulnerability Database (NVD), where disclosed vulnerabilities are catalogued. Each gets a CVE identifier (Common Vulnerabilities and Exposures) — a year plus a number, like CVE-2014-6271 — and a CVSS score from 0 to 10 that tells you how urgently to act. 4 Case study: Shellshock (CVE-2014-6271, CVSS 9.8)

Shellshock scored 9.8 — critical. Why so high? Because it lived in bash (the Bourne-again shell) and could be reached through the Apache HTTP servers that nearly every enterprise runs. To see how it worked, first: what is a shell?

A shell is the command-line interface to the operating system. Applications normally talk to the OS kernel on your behalf, but an administrator can issue commands directly — type ls and the shell asks the OS to list files. Shellshock let an attacker smuggle commands into that shell from the outside, by crafting malicious environment variables in a web request. The result is remote code execution (RCE) — a stranger running commands on your server. Step through it:

How Shellshock dumps your password file

The injected request, in effect, runs:
# attacker smuggles this through a crafted HTTP header
cat /etc/passwd   # read the password file...
# ...and the web server returns it in the HTTP response
A modern echo — Log4Shell
Shellshock's pattern returned in 2021 as Log4Shell (CVE-2021-44228) in the Log4j logging library: another tiny, ubiquitous component, another path to remote code execution, present in countless applications at once. The lesson repeats — a flaw in something everywhere is catastrophic. (Covered on the slides; expand here when you record the Log4j segment.)

5 The vulnerability lifecycle

Every vulnerability moves through stages. The dangerous stretch is the window of exposure — the time a weakness is known to attackers but not yet patched on your systems. Step through it:

From introduced to patched

Responsible disclosure
Good-guy researchers (Google's teams, for example) tell the vendor privately. The vendor builds a patch, and the vulnerability is usually disclosed at the same moment the patch ships — never before, or every attacker would race to exploit it. After that, the clock is on you to patch.

6 Attacker recon vs defender scanning

Your enterprise is a black box to an attacker on the outside — so they probe it, the way a burglar cases a house by checking which doors are unlocked. The defender does the very same probing, first, to themselves. Same tools, opposite intent:

ActivityAttacker (recon)Defender (assurance)
Port scanningFind which ports and services were left open to the world.Confirm no extra ports are exposed; close what shouldn't be open.
ReconnaissanceFingerprint web-server versions and internal structure.Know your own attack surface before someone else maps it.
Vulnerability scanning—Inventory all software; compare versions against the NVD for known CVEs.
Penetration testingExploit a found weakness for real gain.Safely exploit it yourself to prove the gap and fix it first.

The economics matter: cybersecurity is a business on both sides. Whoever finds a vulnerability first — attacker or defender — captures the value, which is exactly why defenders scan relentlessly.

7 Where to look it up

Two public sources put this at your fingertips. The NVD (nvd.nist.gov) is the authoritative catalog. CVE Details (cvedetails.com) is a friendlier front end — browse by year (every disclosure in 2026), filter by vendor (say, all VMware CVEs), and sort by CVSS to surface the 9s and 10s first. Make checking these a habit for any software you run.

Remember

A vulnerability is a weakness; a threat exploits it; risk is the overlap. Weaknesses are configuration, software, or design/architectural (Target was architectural). Disclosed vulnerabilities get a CVE id and a CVSS score (0–10) in the NVD. Shellshock reached bash through Apache to achieve remote code execution — the same shape as Log4Shell years later. Vulnerabilities have a lifecycle, and the gap before you patch is your window of exposure. Attackers and defenders use the same probes; the defender's job is to find the weakness first.
