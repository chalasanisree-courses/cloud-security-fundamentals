Defense in Depth

Module 3 Layered security Segmentation
Learning Objectives
  • Explain why the web, application, and database tiers live on separate subnets
  • Distinguish network-level firewalls from host-level firewalls (AWS security groups)
  • Break a server down into its hardware (CPU, memory, disk, GPU) and its infrastructure → OS → application stack
  • Name a security control at each of the four layers: network, device, application, data
  • Define threat, vulnerability, and risk, and state how they relate

1 A second, closer look

In the previous note we drew the enterprise as a castle: a three-tier app behind a perimeter. That model has one dangerous assumption baked in — that once you're inside the walls, you can roam freely. Real defense doesn't work that way. The fix is defense in depth: not one wall, but many, so that breaching one doesn't hand over everything. Let's zoom in.

2 Split the tiers into subnets

The web, application, and database servers are not on the same network. You separate them into different subnets — for two reasons.

Reason 1: they're different kinds of machines

Take the bank. Every credit-card swipe sends a request that an application server must approve — a flood of small, fast requests. To approve it, the app server fetches your credit profile from a database server — enormous volumes of stored data. One tier is request-heavy; the other is storage-heavy. They're built differently and belong apart.

Reason 2: security

The web server takes requests from the outside and hands valid ones to the app server. The app server — and only the app server — should reach the database. You never want the internet-facing web server touching the database directly.

The rule
Outside → web server (DMZ) → application server → database server. Each hop crosses a subnet boundary, and each boundary is a place to enforce “who may talk to whom.”

3 Two firewalls: network-level and host-level

To enforce those boundaries you use internal firewalls, not just the external one at the perimeter. But there's a subtlety: not every application should reach every database. A network-level firewall controls what travels on the network; a host-level firewall controls what is allowed to reach one specific host or virtual server.

Think of a neighborhood: gates on the streets control which traffic enters the area (network-level), while gates on each house control who reaches that particular door (host-level). Flip the host-level firewall on below and watch the unwanted path get cut:

The host-level firewall — the gate on each individual virtual server — is called a security group. We'll use security groups constantly once we reach the EC2 and VPC modules.

4 What is a server, really?

We keep saying “server.” A server is just a computer — like the one at home, but far more powerful, and built to answer millions of requests from other machines rather than serve one human at a keyboard. Open one up:

Anatomy of a server

Operating system:

Physical or virtual
The hardware doesn't have to be a physical box. In the cloud it's virtual hardware — on AWS, a virtual server is an EC2 instance (Elastic Compute Cloud). Note the rise of the GPU: as AI workloads grow, the CPU offloads heavy processing to the GPU — the shift that took NVIDIA from graphics to the center of computing.

5 The stack: infrastructure → OS → application

On top of the hardware (physical or virtual) sits an operating system — a Linux or Windows server. On top of that run your applications. Security has a job at every level of this stack:

  • Infrastructure / OS — keep the operating system patched and hardened; run antivirus / EDR to watch device and application behavior.
  • Application — the code itself must be written securely, or attackers will exploit its bugs.
  • Communication — the host firewall (security group) ensures the server only talks to the machines it's supposed to.
  • Data — what the application stores must be encrypted (disk encryption), backed up, and access-controlled.

6 Security is multi-tiered

Put it together and security wraps the enterprise in four concentric layers. This is the spine of the entire course — every later module is one of these rings. Click each:

7 Threat vs vulnerability vs risk

One last idea sets up the next note. A vulnerability is a weakness in your system. A threat is an actor trying to exploit it. Neither alone is dangerous — risk lives in the overlap. A fully patched system gives a determined attacker very little to exploit — low risk (never zero: unknown flaws and misconfigurations always exist); an unpatched system with no attacker has no risk yet. Toggle both:

Risk = Threat ∩ Vulnerability

Two worked examples
Misconfigured firewall. You own a firewall but never configured it (vulnerability); an attacker notices and walks through (threat) → risk.   Ransomware. No ransomware protection (vulnerability); an employee clicks a malicious email (threat) → risk: every file in the enterprise is encrypted.
Remember

Defense in depth means layers, not a single wall. Tiers live on separate subnets; network-level firewalls guard the wires while host-level firewalls (AWS security groups) guard each machine. A server is hardware (CPU/memory/disk/GPU) running an OS that must stay patched, running applications that must be coded securely, storing data that must be encrypted. The four rings — network, device, application, data — are the rest of this course. And risk = threat ∩ vulnerability.
