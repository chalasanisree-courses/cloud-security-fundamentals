The CIA Triad

Module 2 Security goals C · I · A
Learning Objectives
  • State the three security goals of the CIA triad and the question each one answers
  • Explain confidentiality through encryption, authentication, and authorization
  • Explain integrity through hashing, digital signatures, and audit trails
  • Explain availability through DDoS mitigation, redundancy, and backup
  • Understand that the three goals trade off against each other, so security is risk-based

1 Why we need information security

Modern systems communicate over networks we don't fully control — above all, the internet. Whenever data is stored or moved, three risks arise naturally:

  • Someone may see information they shouldn't.
  • Someone may change information without authorization.
  • Someone may disrupt access to systems or data.

Security exists to reduce these risks, not eliminate them entirely. Because risks differ by system and context, security decisions are always risk-based.

2 The three goals

The CIA triad names three fundamental security goals. They are objectives, not tools — every security mechanism earns its place by serving one or more of them.

The CIA triad

3 Confidentiality — who can see the data?

Confidentiality ensures that only authorized parties can view information. Three mechanisms work together:

  • Encryption — hides the meaning of data. We saw it secure data in transit (the TLS session); it equally protects data at rest (the account balances sitting in the bank's database should be encrypted too).
  • Authentication — verifies who a user is. A username and password alone is weak authentication (credentials can be stolen); adding a second factor like an SMS code makes it robust — multi-factor authentication (MFA).
  • Authorization — determines what an authenticated user is allowed to access. You should see your account, not another customer's, and never the password database. Controlling access by identity is often done with RBAC (Role-Based Access Control).
Encryption isn't the whole story
Encryption protects the data, but authentication and authorization decide who gets access in the first place. Confidentiality depends on identity and access control as much as on encryption.

4 Integrity — can we trust the data?

Integrity ensures that data has not been altered without detection. It's about the trustworthiness and correctness of data, not secrecy. (“Send $1,000” must not silently become “Send $100,000.”)

  • Hashing — produces a fixed fingerprint of the data; any change yields a different hash, so tampering is detectable.
  • Digital signatures — detect unauthorized modification and bind the data to a specific sender.
  • Audit trails & logging — track changes over time, helping detect unexpected or unauthorized modifications.
Where certificates fit
Integrity is about trusting the data — but that often requires trusting who created or sent it. Digital certificates establish trust in identities. They don't protect data integrity by themselves; they enable trusted integrity checks.

5 Availability — can users reach it?

Availability ensures that systems and data are accessible when legitimate users need them. It's often the most visible security failure — when a site is down, everyone notices.

  • DDoS mitigation — prevents attackers from overwhelming systems with traffic (distributed denial-of-service). Ransomware is a related availability threat.
  • Redundancy — multiple components so there's no single point of failure.
  • Backup & disaster recovery — restore systems and data after a failure or attack.

7 Balance and trade-offs

The three goals frequently conflict:

  • Strong confidentiality controls can reduce availability (more checks, more friction).
  • Strict integrity checks can hurt performance.
  • Maximizing availability may require relaxing certain controls.

There is no perfect configuration. Security is about choosing the right balance based on risk, the system's purpose, and business and user needs — which is exactly what makes it risk-based.

Remember

The CIA triad defines security goals; mechanisms exist to serve them (sometimes more than one). Confidentiality → who can see the data. Integrity → whether the data can be trusted. Availability → whether the system can be used. Every layer in the rest of this course maps back to one or more of these three.
