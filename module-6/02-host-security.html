Securing the Host

Module 6 Host hardening Defense in depth
In this note

1 The network let it in. Now what?

In Module 5 the network layer decided which traffic reaches the instance. But a web server must accept traffic on port 443 — so an attack riding ordinary web requests can sail right past the network. The host has to defend itself.

EC2 is the device / host layer
This is the next ring of defense in depth. The network controls the door; once a request is inside, the instance’s own controls take over. No single layer is meant to be enough on its own.

2 A server is a stack — secure every layer

A server isn’t one thing. It’s a stack: virtual hardware, an operating system, your application — reached over a network interface. Each layer needs its own control. Click a control to see where it applies:

Where each control protects the host

3 Hardening: shrink the attack surface

Zoom in on one layer of the stack — the OS. Every running service is a door an attacker can try. Hardening closes the doors you don’t need. If an instance is only a web server, it has no reason to run FTP — so turn it off:

Disable what you don’t use

Remember

A server is a stack — virtual hardware, operating system, application — reached over a network interface, and every layer needs a control. The Security Group guards the network interface (AWS’s stateful virtual firewall). Patching closes known holes in the OS and application. Hardening shrinks the OS attack surface by disabling unneeded services. Anti-malware watches the running software for what slips through. The virtual hardware itself is AWS’s job. No single layer is enough — together they are defense in depth on the host.
