1 A server you can throw away

An EC2 instance is an on-demand virtual server in AWS. It behaves like any real machine — CPU, memory, storage, networking, an OS — but with one change of mindset: it’s easy to create and easy to destroy.

The instance runs on real physical hardware in an AWS data center, but you never touch that hardware. You declare the performance you need; AWS finds a physical host that fits and places your virtual machine there.

You choose performance — AWS places the machine

Cattle, not pets
Traditional servers were pets — named, nursed, kept alive for years. Cloud instances are cattle: ephemeral, spun up when needed and terminated when not. This single idea drives everything else — scaling, patching by replacement, and where you’re allowed to keep data.

2 Instance types: choosing hardware

You don’t pick a physical server — you pick an instance type, which encodes a hardware class and a size. Decode the name, then pick by workload:

Reading an instance type

Pick a family by workload

Watch the letters
The same letters do different jobs depending on position. As a leading family letter, i = storage-optimized and g = GPU. But the letter after the generation is the processor: in m6i that i means Intel (a = AMD, g = Graviton) — not storage. Read the position, not just the letter.

3 The AMI: your software baseline

The instance type is the hardware. The AMI (Amazon Machine Image) is the software — a template AWS boots from. Click what it carries:

5 Scaling: up vs out

Traffic spikes. You have two ways to cope — make the server bigger, or add more servers. The cloud strongly favors the second:

Vertical vs horizontal scaling

Monitor first
You decide by watching metrics — CPU utilization, network, disk. High CPU on a busy web server is the signal to scale. Scaling out with an Auto Scaling Group adds capacity without disrupting the instances already running.
Remember

An EC2 instance is an on-demand virtual server, built to be ephemeral — cattle, not pets. You choose an instance type (m6i.large = family · generation · processor · size), not the physical hardware. An AMI is the software template — OS, config, and a security baseline — so every launch is identical. Storage is either instance store (fast, local, lost on stop/terminate) or EBS (network-attached, persistent). And you scale either up (bigger instance, needs a restart) or out (more instances via an Auto Scaling Group) — the cloud prefers out.
