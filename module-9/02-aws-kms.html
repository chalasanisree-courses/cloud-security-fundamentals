Keys in the Cloud: AWS KMS

Module 9 KMS · HSM Key policies
In this note

1 The same idea, at cloud scale

Everything from the last note — vaults, envelope encryption — AWS runs as a service. AWS KMS (Key Management Service) manages your keys for you and is the front door to the underlying HSMs.

How S3 encryption really works
When S3 encrypts an object, it doesn’t hand your data to the HSM. It asks KMS for a data key (DEK), encrypts the object with it in the service, then stores the wrapped DEK beside the object. Your KMS key (the KEK) never leaves the HSM — exactly the envelope pattern, run for millions of objects.

3 The access twist: a key is not a bucket

Keys are too sensitive to leave to ordinary permissions. So a key policy is mandatory on every KMS key — and access works differently from an S3 bucket. Compare the two:

S3 (one gate, either side) vs KMS (the key policy rules)

The precise rule (so you don’t over-simplify)
The key policy is the root of trust for a key and is always required. The common default key policy delegates to IAM (the “Enable IAM permissions” statement), so in practice a user needs both the key policy to permit IAM and an IAM policy granting the action — hence “two gates.” But a key policy can also grant a principal directly with no IAM policy. AWS evaluates the key policy, IAM policies, and grants together (not in sequence), with an explicit deny always winning.
Remember

AWS KMS is the managed key service and the front door to the HSMs that hold your keys; S3 and friends use it for envelope encryption — KMS hands out a wrapped DEK while the KMS key (KEK) stays in the HSM. Keys live in either AWS-managed multi-tenant HSMs (the default) or a CloudHSM single-tenant cluster you run for strict isolation. Every key carries a mandatory key policy. Unlike an S3 bucket — where an IAM policy or a bucket policy can grant access — a KMS key’s key policy is the root of trust: it must permit the access (directly, or by delegating to IAM), so you typically need both the key policy and an IAM policy, with explicit deny always winning.
