Enterprise Trends

Module 1 Foundations Virtualization & Cloud
Learning Objectives
  • Describe the traditional corporate network and the boundary that protects it
  • Explain virtualization and how the hypervisor consolidates many servers onto one machine
  • Distinguish IaaS from SaaS by who is responsible for what
  • Articulate the shared‑responsibility gradient that runs from traditional on‑prem to SaaS

1 Where we are

So far we've watched the consumer internet grow — Dropbox, Gmail, Amazon, social apps delivered from the cloud. This lecture turns to what was happening inside companies, because two enterprise shifts — virtualization and cloud computing — are what made modern cloud security a discipline at all. They also introduce the single most important idea for the rest of the course: shared responsibility.

2 The traditional corporate network

Picture a company — say General Electric. Employees get client devices (laptops, desktops) and use them to reach applications running on servers the company owns:

  • A salesperson takes an order on their laptop → the order is stored on a database server;
  • an application server processes it;
  • the customer checks status through a web server.

The traditional corporate network

Around all of this sits a strong network boundary: firewalls and security appliances separate the corporate network from the internet. A few machines (web servers) face the internet; the rest of the network stays isolated.

The data center was a maintenance burden
Running your own servers meant rooms full of racks, each typically running a single application. That demanded heavy cabling to interconnect racks, backup power so applications stayed available, and cooling for the heat all that hardware produced.

3 Shift one — virtualization

By the 2000s, CPUs and memory had become so powerful that one‑app‑per‑server left machines badly underutilized. The fix was the hypervisor — a virtualization layer that lets a single physical server present itself as many virtual servers, each with its own OS and apps. Pioneered by VMware, it let companies consolidate many machines into few — cutting cost, cabling, power, and cooling.

One physical server: before vs after virtualization

4 Shift two — cloud computing

In the 2010s, companies like AWS (Amazon Web Services) filled huge data centers with servers and let other companies simply rent them over the internet — no hardware to own. It's like calling an Uber when you need a ride instead of buying and maintaining a car: you get the capability without owning the asset. The key insight: a responsibility gradient
Traditional — you run the infrastructure and the apps. IaaS (renting an AWS server) — the provider runs the physical infrastructure; you still run and maintain the applications on top. SaaS (Office 365 instead of running Exchange yourself) — you don't worry about infrastructure at all; you just use the application.

5 Why virtualization made cloud possible

◈ Multi-tenancyMany customers, one machine, isolated ▶

Inside AWS, one physical machine can host isolated virtual servers for many different customers — GE and Coca‑Cola side by side, walled off from each other. Without virtualization, renting out infrastructure at this scale wouldn't be practical.

◈ ElasticitySpin up a server in seconds ▶

Giving a customer another server no longer means buying and racking hardware — it's near‑instant. That on‑demand provisioning is the practical magic of the cloud, and it's only possible because the servers are virtual.

Remember
Traditional → IaaS → SaaS is a hand‑off of responsibility, not of accountability. The more the provider manages, the less you control — but security is always shared. This gradient underlies every layer in the rest of the course.
