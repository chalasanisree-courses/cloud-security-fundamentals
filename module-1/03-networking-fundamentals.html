1 Where we are

The previous two notes covered why the internet matters — its history, and the shift to enterprise and cloud computing. This note is the gear‑change into how data actually moves — the mechanics every later security topic builds on.

The internet took off because of two standard protocols: TCP (Transmission Control Protocol) and IP (Internet Protocol). This lecture follows one ordinary action — typing example.com into a browser — all the way down the TCP/IP stack and across the internet to a server, then back. Understanding this path is the prerequisite for everything we secure later: firewalls, VPCs, TLS, and more all operate on the mechanics below.

The client‑server model
Your device acts as a client: it goes out over the internet, fetches the page from the web server that hosts example.com, and loads it. Everything below explains how that round trip actually happens. And note: right now this channel is completely open — nothing is encrypted — which is exactly what Module 2 sets out to fix.

2 IP routing is like the postal service

IP routing works hop by hop. No single point knows the whole path — each stop only knows its next hop. Sending a letter from San Jose to Sofia, Bulgaria, every facility that doesn't know how to reach the destination simply forwards it to the next stop up the chain, until one does. Step through it:

Packet journey — next-hop forwarding

The crucial detail
At every hop the destination IP stays the same (it's the final address), but the MAC address is rewritten to point at the next machine in the chain. IP says where it's ultimately going; MAC says the very next physical step.

3 What's inside the packet — and DNS

Routing uses numbers, not names, so your browser first does a DNS resolution (Domain Name Service) to turn example.com into an IP address:

Why this matters for security
Everything downstream trusts this name→IP answer. Poison it and users are silently sent to an attacker — a DNS spoofing attack, where the lookup returns a bad IP address. We cover it in Module 2.

The destination MAC is the unknown: the destination isn't on your local network, so the packet goes to the next hop (your router) and the MAC is rewritten at every hop — the switching and ARP story in Section 9.

4 Finding your own addresses

You can inspect your machine's addresses from a terminal — ipconfig on Windows, ifconfig on macOS/Linux:

Terminal — ifconfig (macOS)
en0:  (Wi-Fi interface)
  ether   a4:83:e7:2b:6d:1f          # MAC address (physical, fixed)
  inet    192.168.1.165             # IPv4 address (given by your router)
  inet6   fe80::1c3a:...             # IPv6 address

Every device on your home network shares the same network prefix and gets a different host number: your Mac 192.168.1.165, your phone 192.168.1.250, the router 192.168.1.254.

IP routes across networks; MAC routes within one
Leave home and join the Starbucks Wi‑Fi: your MAC stays the same (it's the physical address of your network card), but you get a new IP from the Starbucks router (assigned automatically by DHCP, the Dynamic Host Configuration Protocol). MAC = which device on this local network. IP = how to cross between networks. (See it live: switch your laptop to a phone hotspot — the phone becomes the router and hands out a new IP, while your MAC is unchanged.)
“MAC” here = Media Access Control
A MAC address is the fixed physical address burned into a device's network card — Media Access Control. It has nothing to do with an Apple Mac computer, even though the demo happens to use one. MAC addresses operate at the Network Access layer (TCP/IP) / Data Link layer, Layer 2 (OSI).

5 IPv4 vs IPv6, and NAT

Address space

Because IPv4 ran short, your home router performs NAT (Network Address Translation): every device on your private network shares one public IP on the way out. Load showmyip.com and you'll see something like 104.56.116.217 — not your private 192.168.x.x. NAT exists for two reasons: it conserves scarce IPv4 addresses (many devices behind one public IP), and it adds a measure of security by hiding your internal addresses.

6 The transport layer — the handshake

Before any web page is fetched, the client and server must agree to talk. That's the TCP three‑way handshake:

Three-way handshake

TCP and IP: a division of labor
IP delivers, but doesn't guarantee. IP is connectionless and unreliable by itself — it just forwards packets hop by hop, with no promise they arrive, arrive intact, or arrive in order. TCP guarantees, but doesn't move packets. Riding on top of IP, TCP adds the session (the handshake), correct ordering, and retransmission of anything lost. IP moves the packets; TCP makes the conversation reliable.

Not everything needs TCP's guarantees. UDP (User Datagram Protocol) is the fast, connectionless alternative — no handshake, no retransmission. DNS lookups, video streaming, and voice calls use UDP, where speed matters more than perfect delivery.

9 Routing vs switching: Layer 3 and Layer 2

The hop‑by‑hop journey earlier happens across networks, using IP addresses. That's routing — the IP protocol, at the Internet layer (TCP/IP) / Network layer, Layer 3 (OSI).

But two devices on the same local network — your computer and a smart TV on the same Wi‑Fi — don't need to cross any network. Delivery is by MAC address using the Ethernet protocol. That's switching, at the Network Access layer (TCP/IP) / Data Link layer, Layer 2 (OSI).

OperationUsesProtocolLayer
Routing — across networksIP addressIPLayer 3 (Internet / Network)
Switching — within the local networkMAC addressEthernetLayer 2 (Network Access / Data Link)

How does a device learn a neighbor's MAC? ARP

To switch a frame to the TV, your computer needs the TV's MAC — but it only knows the TV's IP. It discovers the MAC with ARP (Address Resolution Protocol): it broadcasts to everyone on the subnet asking “who has this IP?”, and the matching device replies with its MAC. Step through it:

ARP — finding a neighbor's MAC

One nuance: if the destination is on another network, your computer can't ARP for it — the destination isn't local. Instead it ARPs for the default gateway (your router) and hands the packet there. So ARP always resolves the next local hop: the destination device when it's local, or the gateway when the destination is remote.

Reporting problems: ICMP

Not every packet carries application data. ICMP (Internet Control Message Protocol) is a Layer 3 protocol for control and error messages about IP communication — carried inside IP packets, so it travels across routers and networks. It powers two everyday tools: ping (an ICMP echo request and reply — “can I reach this host?”) and traceroute (which reads ICMP time‑exceeded messages to map each hop along the path). ICMP carries no application data; it just lets devices report conditions and verify connectivity.

ARP vs ICMP — don't confuse them
ARP prepares an IP packet for local delivery (find the MAC for the next hop). ICMP is an IP‑level message that reports network conditions (ping, traceroute). One resolves addresses; the other reports status.

10 Subnets and CIDR

A subnet mask says how many bits of an address are the network portion and how many are the host portion. A mask of 255.255.255.0 means the first 24 bits are network and the last 8 are host — written in CIDR as /24. That's why every device on the home network shares 192.168.1. and only the last number changes.

An IPv4 address is 32 bits (four 8‑bit octets). Host bits = 32 − prefix. Drag the slider to explore — this recreates the cidr.xyz demo:

Remember

Establish a session first (the TCP handshake), then route packets hop by hop (IP), resolve names with DNS, and deliver within the local network by MAC. TCP = a reliable conversation; IP = step‑by‑step delivery.
