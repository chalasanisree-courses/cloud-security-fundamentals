1 The data layer

We’ve secured how traffic reaches systems (network) and how the servers themselves are hardened (device). This layer asks a different question: where does the application’s data live, and how do we protect the data itself?

Why a separate layer
Network and host controls guard the paths to data. The data layer guards the data itself — so that even if a server or network control fails, the stored information is still defended. In AWS, the core data-layer service is Amazon S3.

2 Block, file, or object?

Applications pick storage by how they need to use it. AWS offers three shapes — and the difference is the unit each one stores. Click through:

Three kinds of cloud storage

The one-line distinction
EBS works with blocks, EFS works with files, S3 works with objects. EBS is a disk for one server; EFS is a shared filesystem several servers mount; S3 is an API you send whole objects to.

3 S3 is a shared kitchen’s refrigerator

The cleanest way to picture S3: AWS is a shared commercial kitchen. Click each piece to see what it maps to:

The shared-kitchen analogy

S3 lives outside your VPC
The refrigerator isn’t inside any one kitchen — it’s shared infrastructure the whole facility uses. Likewise S3 is a regional service that lives outside your VPC; your apps reach it over the S3 API, not by mounting a disk. (A VPC endpoint can give a private path to S3, but it never puts S3 inside your VPC.)
Remember

The data layer protects the data itself, not just the paths to it — in AWS, that’s Amazon S3. Storage comes in three shapes by unit: block (EBS, a disk for one server), file (EFS, a shared filesystem), and object (S3, whole objects via an API). S3 keeps objects inside buckets and has no filesystem to mount. And like a shared refrigerator in a commercial kitchen, S3 is shared infrastructure outside your VPC, reached over its API.
