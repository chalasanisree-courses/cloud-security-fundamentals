1 The IAM identities

AWS gives you a small set of identity building blocks. Click each to see what it is and when to use it:

Root, users, groups, roles

Two things students always miss
A group is not an identity — it’s just a bucket of users you attach policies to; it can never be a Principal or make a request itself. And a role has no password — you assume it for short-lived credentials, which is how AWS does federation.

3 How AWS decides: deny wins

When you make a request, AWS gathers every policy that applies and evaluates them together by one rule. Run requests through it:

Explicit deny → explicit allow → implicit deny

Least privilege
Because access is denied unless granted, you build up from nothing — give each identity only the permissions it actually needs. That’s the principle of least privilege, and an explicit deny is the unbreakable backstop on top of it.

4 The AWS twist

AWS handles the three pillars differently from the traditional model. Click each to see the shift:

Traditional model vs the AWS way

Remember

IAM’s identities are the root user (all-powerful — lock it away), IAM users (long-lived people), groups (containers for users, never a principal), and roles (temporary, assumed for short-lived credentials — AWS’s federation mechanism). Permissions are JSON policies: identity-based (attached to a user/group/role, no Principal) and resource-based (attached to a resource, with a Principal). AWS evaluates all of them together: an explicit deny always wins, otherwise you need an explicit allow, and silence is an implicit deny. The AWS twist: identities are ephemeral roles, access is resource-level (no general admission), and one unified policy covers both entry and actions.
