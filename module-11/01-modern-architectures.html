1 How we run code: four steps

Computing keeps handing more of the stack to the cloud vendor. Each step packs apps tighter and leaves you managing less. Step through them — the shipping analogy rides along:

Physical → VM → container → serverless

Why containers exist
Two apps that need different library versions (Python 3.12 vs 3.14) used to need separate machines — “dependency hell.” A container packages an app with its own libraries, so many isolated apps share one machine. That solves both the dependency clash and the waste of a whole OS per app.

3 Serverless, and the new front door

At the far end of the evolution, you stop managing servers entirely — and with no servers to balance, the load balancer gives way to an API Gateway. Compare the two serverless flavors:

Fargate vs Lambda · and LB → API Gateway

The front-door swap
A load balancer spreads requests across servers. In serverless there are no servers, so an API Gateway takes its place — it validates each API request and triggers the right Lambda function to handle it. Same job (the front door), different world.
Remember

Two evolutions run in parallel. How we run code: physical servers → VMs (one OS per app, wasteful) → containers (package an app with its libraries, pack many on one machine, end dependency hell) → serverless (the vendor runs the OS and runtime; you bring only code). How we structure code: a monolith (everything in one program) → microservices (independent services that scale and ship on their own). Serverless comes in two forms — Fargate (you hand over a container) and Lambda (you hand over a function) — and because there are no servers to balance, the load balancer becomes an API Gateway.
