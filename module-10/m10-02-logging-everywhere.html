Visibility Across Every Layer

Module 10 Cross-cutting Log sources
In this note
  • Why logging isn’t a layer — it’s visibility that cuts through every layer
  • The log source for each layer: identity, network, device, application, data
  • Why a gap in one layer’s logs becomes a blind spot

1 Logging is vertical, not horizontal

Every other module so far has been a layer of defense in depth — network, device, data, identity. Logging is different: it’s not one of the rings, it’s the visibility that runs through all of them. Each layer keeps its own kind of log. Click a layer:

One log source per layer

2 A gap in one layer is a blind spot

The layers only give you the full story together. Drop one and you lose the answer it held. Follow a single suspicious request and toggle whether you kept the application logs:

Network logs alone aren’t enough

The takeaway
No single log tells the whole story. With network logs but no application logs, you can see traffic reached the server but have no idea whether it errored, was rejected, or quietly succeeded. Real investigations stitch layers together — which is why you collect them all.
Remember

Logging is not a layer — it’s the vertical visibility that cuts through every layer of defense in depth. Each layer has its own log source: identity → CloudTrail (who signed in, who assumed a role), network → VPC Flow Logs (traffic to your interfaces), device → system logs (who ran sudo), application → app logs (NGINX 404s and errors), data → S3 access logs (who touched the files). You need them all: a gap in one layer is a blind spot — network logs without application logs show traffic arriving but never tell you what the server did with it.
